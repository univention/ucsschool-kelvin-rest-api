# Task 005 — Migration: fixed `pg_trgm` GIN indexes

**Status:** not started

## Objective

Add a new Alembic migration that enables the `pg_trgm` Postgres extension and
creates GIN trigram indexes on the 6 fixed columns identified for indexed
case-insensitive wildcard search: `user.name`, `user.firstname`,
`user.lastname`, `user.email`, `school.name`, `group.name`.

## Context

See [`../context.md`](../context.md) for full background on the DB layer and
Alembic setup. Relevant decisions: [`../decisions.md`](../decisions.md) D2
(which columns), D3 (GIN trigram only, no `lower()` tier), D6 (raw SQL DDL
style, plain `CREATE INDEX`, no ORM model changes).

Alembic chain today: `f1c5bf519a40` → `e8b27dd51414` → `a3f9c12e8b01` →
`62810ee19208` (current head, "rename_group_type_to_role"). This new
migration's `down_revision` must be `"62810ee19208"`.

No Postgres extensions or GIN/GIST indexes exist anywhere in the codebase
today. Reference style for a migration file:
`alembic/versions/f1c5bf519a40_init_tables.py`.

## Scope

- One new file: `alembic/versions/<new_revision>_add_trigram_indexes.py`.
- `CREATE EXTENSION IF NOT EXISTS pg_trgm`.
- `CREATE INDEX IF NOT EXISTS ix_<table>_<col>_trgm ON "<table>" USING gin
  ("<col>" gin_trgm_ops)` for each of the 6 columns.
- `downgrade()` drops the 6 indexes; does **not** `DROP EXTENSION pg_trgm`.

## Non-goals

- UDM property indexes — that's Task 007, a separate migration.
- Any change to `ucsschool-objects/src/ucsschool_objects/database_models.py`
  (per D6 — migration-only DDL, no ORM `Index()` objects, since
  `Base.metadata` is shared with the SQLite test fixtures and `gin`/
  `gin_trgm_ops` are Postgres-only).
- `CREATE INDEX CONCURRENTLY` — documented as a fallback option in this
  task's file, not implemented by default (see Risk 3 in `../context.md`).

## Dependencies

None — can start immediately (Phase A).

## Implementation steps

1. Generate a new revision id (or hand-pick one following the existing
   12-character hex style used in this repo, e.g. via
   `python -c "import uuid; print(uuid.uuid4().hex[:12])"` or Alembic's own
   `revision` command if a dev DB is available:
   `make alembic-migration ALEMBIC_MESSAGE="add trigram indexes"` then adapt
   the autogenerated skeleton, since autogenerate won't detect
   extension/GIN-index needs on its own).
2. Write `upgrade()`:
   ```python
   from alembic import op
   import sqlalchemy as sa

   revision = "<new>"
   down_revision = "62810ee19208"
   branch_labels = None
   depends_on = None

   _INDEXES = [
       ("ix_user_name_trgm", "user", "name"),
       ("ix_user_firstname_trgm", "user", "firstname"),
       ("ix_user_lastname_trgm", "user", "lastname"),
       ("ix_user_email_trgm", "user", "email"),
       ("ix_school_name_trgm", "school", "name"),
       ("ix_group_name_trgm", "group", "name"),
   ]

   def upgrade() -> None:
       op.execute(sa.text("CREATE EXTENSION IF NOT EXISTS pg_trgm"))
       for index_name, table, column in _INDEXES:
           op.execute(
               sa.text(
                   f'CREATE INDEX IF NOT EXISTS {index_name} '
                   f'ON "{table}" USING gin ("{column}" gin_trgm_ops)'
               )
           )
   ```
3. Write `downgrade()`:
   ```python
   def downgrade() -> None:
       for index_name, _table, _column in _INDEXES:
           op.execute(sa.text(f"DROP INDEX IF EXISTS {index_name}"))
       # Intentionally not dropping the pg_trgm extension: Task 007's
       # dynamic UDM-property indexes may depend on it, and migrations
       # run linearly, so dropping a shared extension here is unsafe.
   ```
4. Add a short module docstring noting: plain `CREATE INDEX` (not
   `CONCURRENTLY`) is used deliberately (see D6/Risk 3 in `../context.md`) —
   this briefly locks writes to the affected tables; deployments with very
   large `user`/`school`/`group` tables should evaluate switching to
   `CREATE INDEX CONCURRENTLY` via `op.get_context().autocommit_block()`
   before running this in production, and validate that against the
   advisory-lock context manager in `alembic/env.py`.

## Acceptance criteria

- `alembic upgrade head` successfully creates the extension and all 6
  indexes on a fresh Postgres database.
- `alembic downgrade -1` successfully drops the 6 indexes without erroring
  (and without dropping the extension).
- Running `upgrade` twice (idempotency check) does not error, thanks to
  `IF NOT EXISTS`.
